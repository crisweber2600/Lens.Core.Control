# S5.1 — Scaffold StoreOperationsService .NET Project

## Context

The `StoreOperationsService.Domain` project and test project already exist. This story adds the runnable API layer: a minimal .NET Web project with a `/health` endpoint and the Aspire-compatible health-check wiring. This is the surface S5.2 will register in the AppHost.

**Target repo:** `TargetProjects/coffee/store-service/coffee3`
**Solution file:** `TargetProjects/coffee/store-service/coffee3/StoreOperationsService.sln`

## User Story

**As a** developer,  
**I want** a runnable `StoreOperationsService.Api` project scaffolded in the coffee3 solution,  
**so that** S5.2 can register it in the shared Aspire AppHost and E1/E3 stories have a valid HTTP surface to bind to.

## Acceptance Criteria

- [ ] `src/StoreOperationsService.Api/StoreOperationsService.Api.csproj` exists
- [ ] Project is .NET 10 (`net10.0` TFM), targeting `Microsoft.NET.Sdk.Web`
- [ ] `StoreOperationsService.Api` references `StoreOperationsService.Domain`
- [ ] `Program.cs` registers health checks and maps `GET /health` returning 200 OK
- [ ] Project added to `StoreOperationsService.sln` in the `src` solution folder
- [ ] `dotnet build StoreOperationsService.sln` succeeds with no errors (nullable warnings acceptable)
- [ ] `dotnet test` still passes (29/29 existing tests green)
- [ ] Integration test `StoreOperationsService.Tests` includes a `HealthCheckTests.cs` that boots the API via `WebApplicationFactory<Program>` and asserts `GET /health` returns 200

## Technical Notes

- Use `app.MapHealthChecks("/health")` with `builder.Services.AddHealthChecks()`
- Keep `Program.cs` minimal — no business logic
- Aspire resource binding will be added in S5.2; do not add Aspire SDK yet
- Export `Program` as `public partial class Program {}` (or `internal` with `InternalsVisibleTo`) so `WebApplicationFactory` works in tests
