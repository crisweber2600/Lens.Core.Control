# S1.1 — Define StoreOrder Aggregate and Value Objects

## Context

This is the foundational story for the entire StoreOperationsService. All other E1, E3, and E4 stories depend on this aggregate contract. The goal is to define the domain model — no persistence yet.

Key architecture decisions from `architecture.md` that this story implements:

- **AD-1:** Single store-order aggregate with relational current-state + append-only log + transactional outbox. The domain model must be designed to support all three persistence layers.
- **AD-2:** Lifecycle state is separate from operational modifiers (rush, at-risk). The value objects must reflect this separation.
- **AD-3:** `completed` means handoff confirmed (operational moment), not CustomerService-driven.
- **AD-4:** `completed → cancelled` is an invalid transition and must throw at the aggregate boundary.

## User Story

**As a** developer,  
**I want** a `StoreOrder` aggregate root and its full domain model defined,  
**so that** all lifecycle stories have a consistent and tested domain contract to build on.

## Acceptance Criteria

- [ ] `StoreOrder` class with properties: `OrderId` (Guid), `LifecycleState` (enum), `OperationalModifiers`, `CreatedAt`, `UpdatedAt`, `CustomerId` (from CustomerService event)
- [ ] `OrderLifecycleState` enum: `Received`, `Queued`, `InProgress`, `Ready`, `Completed`, `Cancelled`
- [ ] `OperationalModifiers` value object with: `IsRush` (bool), `IsAtRisk` (bool), `PriorityBand` (enum: Standard, Rush)
- [ ] `StoreOrder.Transition(newState)` method that validates against the transition table
- [ ] `InvalidTransitionException` thrown for invalid transitions — message includes `orderId`, `fromState`, `toState`
- [ ] `Completed → Cancelled` specifically unit tested and throws (AD-4)
- [ ] All valid transitions unit tested
- [ ] All invalid transitions unit tested
- [ ] No persistence dependencies — domain model only (no DbContext, no repositories)
- [ ] Code is in the appropriate domain layer / project namespace

## Test Scenarios

```
Given a StoreOrder in state Received
When Transition(Queued) is called
Then state is Queued and no exception is thrown

Given a StoreOrder in state Completed  
When Transition(Cancelled) is called
Then InvalidTransitionException is thrown with message containing "Completed → Cancelled"

Given a StoreOrder in state InProgress
When Transition(Ready) is called
Then state is Ready

Given a StoreOrder in state Cancelled
When any Transition is called
Then InvalidTransitionException is thrown (terminal state)
```
