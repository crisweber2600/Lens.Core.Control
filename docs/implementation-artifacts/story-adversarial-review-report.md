## Per-Story Reviews

## Story 1.1 — New Codebase Scaffold, Install Surfaces, and Module Surface Reduction
### Adversarial Review
- **Finding 1 — the story under-specifies discovery parity.** The story file reduces the requirement to “17 published prompt stubs” plus aligned `setup.py`, `module-help.csv`, `lens.agent.md`, and adapters, but the source-of-truth backlog requires parity across **prompt stubs, help surfaces, manifests, and adapters everywhere the user can invoke Lens**. It also explicitly requires `split-feature` to remain visible on every published discovery surface. The story therefore omits explicit coverage for manifests and broad “every discovery surface” parity, which is material for acceptance. (`docs/lens-dev/new-codebase/lens-dev-new-codebase-baseline/stories/1-1-scaffold-published-surface.md:20-28`, `docs/lens-dev/new-codebase/lens-dev-new-codebase-baseline/stories.md:49-61`, `docs/lens-dev/new-codebase/lens-dev-new-codebase-baseline/prd.md:109-113`)
- **Finding 2 — alias and stale-advertising behavior is missing.** Epic 1.1 requires that supported adapters expose the same 17-command set and that command aliases resolve to the same public names as the prompt stubs; it also requires that no removed public stub is still advertised. The story mentions supported adapters, but not alias resolution or stale discovery cleanup, so it is weaker than the accepted epic contract. (`docs/lens-dev/new-codebase/lens-dev-new-codebase-baseline/epics.md:203-216`, `docs/lens-dev/new-codebase/lens-dev-new-codebase-baseline/architecture.md:94-96`)
- **Finding 3 — the story treats inventory as a checklist, but architecture treats it as a runtime safety gate.** The story says to audit the retained internal skill inventory against the architecture matrix and “record any remaining drift before merge.” The architecture is stricter: retained commands must keep the full chain from prompt stub -> release prompt -> owning skill, and internal skills required by retained commands must survive even when public stubs are removed. That makes the inventory load-bearing, not advisory. (`docs/lens-dev/new-codebase/lens-dev-new-codebase-baseline/stories/1-1-scaffold-published-surface.md:34-37`, `docs/lens-dev/new-codebase/lens-dev-new-codebase-baseline/prd.md:137-141`, `docs/lens-dev/new-codebase/lens-dev-new-codebase-baseline/architecture.md:85-96,147-156,218-234`)
- **Finding 4 — the story title still reflects reduction more than parity.** Current source-of-truth docs rename the outcome to “Stable Published Surface and Discovery Parity,” emphasizing coherence and retention rather than simple reduction. The current story title is not wrong, but it is narrower than the approved framing and can bias implementation toward deletion instead of parity proof. (`docs/lens-dev/new-codebase/lens-dev-new-codebase-baseline/stories/1-1-scaffold-published-surface.md:14`, `docs/lens-dev/new-codebase/lens-dev-new-codebase-baseline/stories.md:49-61`, `docs/lens-dev/new-codebase/lens-dev-new-codebase-baseline/epics.md:195-216`)
- **Finding 5 — clean-room posture is corrected, but the story still needs stronger release-gate wording.** FinalizePlan confirms the story was corrected to use old-codebase docs as verification-only references, not implementation inputs, which is good. But FinalizePlan also flags a release-level integration/coherence gap: parity checks must prove cross-command coherence, not only per-work-package success. Story 1.1 does not yet call out that its discovery-surface parity evidence must be consumable by the later release gate. (`docs/lens-dev/new-codebase/lens-dev-new-codebase-baseline/stories/1-1-scaffold-published-surface.md:41-43`, `docs/lens-dev/new-codebase/lens-dev-new-codebase-baseline/finalizeplan-review.md:41-45,193-196`)

## Story 1.3 — Implement bmad-lens-batch Shared 2-Pass Contract

### Adversarial Review
**Strengths**
