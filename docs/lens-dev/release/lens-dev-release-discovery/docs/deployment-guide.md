## Distribution Model

`lens.core.release` is distributed as a **git repository**, versioned by tag and published on the `beta` branch. Consumers install it by cloning and running the setup script — there is no package registry (no npm, no PyPI, no NuGet).

```
[lens.core.src source repo]
       ↓  (promotion pipeline)
[lens.core.release beta branch]
       ↓  (git clone)
[consumer control repo: lens.core/]
       ↓  (install.py)
[IDE adapters: .github/, .claude/, .codex/, .cursor/]
```

---

## Versioning

Version identity is declared in `_bmad/lens-work/module.yaml`:

```yaml
module:
  version: "4.0.0"
  schema: 4
```

- `version`: Semantic version string
- `schema`: Integer schema version used for migration checks

Consumers validate version compatibility during preflight. Schema migrations via `/lens-migrate` and `/lens-upgrade` handle version transitions.

## Release Branches

| Branch | Purpose |
| --- | --- |
| `beta` | Current stable release — what consumers clone |
| `main` | Long-term stable |
| `v3.x` | Previous schema series (legacy) |

Consumers always pin to `beta` for the latest release:
```bash
git clone --branch beta https://github.com/crisweber2600/lens.core.git
```

## Consumer Install Flow

### Minimum install
Clone the module into the consumer control repo root:
```bash
cd /path/to/my-control-repo
git clone --branch beta https://github.com/crisweber2600/lens.core.git
```

The module sits at `./lens.core/` relative to the control repo root.

### Bootstrap and configure
```bash
uv run lens.core/_bmad/lens-work/scripts/setup-control-repo.py
```

This generates:
- `lens.core/bmadconfig.yaml` — Consumer configuration (GitHub org, username, etc.)
- `.lens/` — Personal state directory (git-ignored)
- `.github/`, `.claude/`, `.codex/`, `.cursor/` — IDE adapters (via `install.py`)

### IDE adapter generation
Adapters are generated by `install.py` from `_module-installer/`:
```bash
uv run lens.core/_bmad/lens-work/scripts/install.py
```

Each IDE adapter is a thin stub that:
1. References `light-preflight.py` from the cloned module
2. Routes to the SKILL.md content inside `lens.core/_bmad/lens-work/skills/`

Adapters are idempotent — safe to regenerate at any time.

## Upgrading

To pick up a new release:

```bash
cd lens.core
git pull origin beta
```

Then re-run the install to refresh IDE adapters:
```bash
uv run lens.core/_bmad/lens-work/scripts/install.py
```

If the schema version incremented, run the upgrade or migrate flow:
```
/lens-upgrade
```
or
```
/lens-migrate
```

## Governance Repo Topology

The governance repo is a **sibling or submodule** of the control repo. It stores planning artifacts and feature YAML, separate from code.

```
TargetProjects/
├── {project-name}.src/         # Control repo (code)
└── {project-name}.governance/  # Governance repo (plans)
    └── features/
        └── {domain}/
            └── {service}/
                └── {feature}/
                    ├── feature.yaml
                    ├── businessplan.md
                    ├── techplan.md
                    └── ...
```

The governance repo name is derived from the control repo root folder name, appended with `.governance` (e.g., `printingmagic.src` → `printingmagic.src.governance`).

## Parity Validation

To verify the release distribution matches the source:
```bash
bash _bmad/lens-work/scripts/validate-lens-core-parity.sh
```

This checks that all skills, prompts, and scripts are present in both source and release.

## Two-Branch Topology at Deploy Time

Each new feature creates two git branches in the governance repo:

| Branch | Prefix | Content |
| --- | --- | --- |
| Plan branch | `{domain}-{service}-{featureSlug}-plan` | All planning artifacts |
| Dev branch | `{domain}-{service}-{featureSlug}` | Target-repo working refs |

Both branches live in the governance repo. Target repos create short `feature/{featureSlug}` working branches. This separation keeps planning clean from code.
