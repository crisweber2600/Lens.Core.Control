## Important: Release vs. Source

`lens.core.release` (this repo) is a **read-only distribution artifact**. It is the promoted output of the source repository. Do not submit pull requests or make direct edits to this repo.

```
[lens.core.src]   ←  ALL contributions go here
       ↓  promotion
[lens.core.release]   ←  read-only distribution
```

To contribute to the LENS Workbench module, work in **`lens.core.src`**.

## Promotion Pipeline

Changes flow from source to release through a promotion process:

1. **Author** changes in `lens.core.src` (source)
2. **Test** locally using `uv run --with pytest pytest _bmad/lens-work/scripts/tests -q`
3. **Validate** registry with `uv run _bmad/lens-work/scripts/validate-lens-bmad-registry.py`
4. **Create PR** from feature branch to source `main`
5. **CI** runs parity check: `bash _bmad/lens-work/scripts/validate-lens-core-parity.sh`
6. **Promote** — source main is promoted to release `beta` branch via release pipeline

## Development Conventions in lens.core.src

### Skill additions
- Every new skill needs a corresponding `lens-{name}.prompt.md` in `prompts/`
- Register the skill in `module.yaml` under `skills:`
- Add the command to `module-help.csv`
- Update `agents/lens.agent.md` shell menu if it is a user-facing command
- See repo memory: "Lens command discovery surfaces" — all three discovery files must be updated in sync

### Script additions
- Use inline `uv` script headers for dependencies:
  ```python
  # /// script
  # dependencies = ["pyyaml>=6.0"]
  # ///
  ```
- Write a corresponding test file in `scripts/tests/`
- Use `uv run --with pytest pytest` as the test runner pattern

### Prompt changes
- Prompts follow a two-line pattern:
  1. Run `light-preflight.py`
  2. Load `SKILL.md` and follow instructions
- Prompt stubs in `.github/prompts/`, `.claude/`, etc. are generated — edit the source at `_bmad/lens-work/prompts/`

### Lifecycle changes
- Schema-breaking changes increment `schema:` in `module.yaml`
- New phases or transitions must be declared in `lifecycle.yaml`
- Breaking schema changes require a `migrate-ops.py` migration path

## Reporting Issues

If you encounter a bug in the release distribution:

1. Verify you are on the latest `beta` release: `git -C lens.core pull origin beta`
2. Check if the issue reproduces with the latest source via the source repo
3. File an issue in the source repository, not the release repository

Feature requests and improvement proposals follow the LENS Workbench feature lifecycle:

1. Create a governance feature in your control repo: `/lens-init-feature`
2. Scope the idea with `/lens-preplan`
3. Write a business case with `/lens-businessplan`
