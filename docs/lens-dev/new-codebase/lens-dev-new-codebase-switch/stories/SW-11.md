# SW-11 — Document JSON Contracts

## Context

The switch skill reference must document all response shapes so downstream callers (release prompt, dev agents, test authors) can rely on a single source of truth for the API contract. Undocumented contracts lead to callers guessing at field names or handling errors incorrectly.

## Task

Update `lens.core/_bmad/lens-work/skills/bmad-lens-switch/SKILL.md` (or a new `references/contracts.md`) to document all four JSON response shapes:

1. **List success — features mode**: all fields including `num`, `id`, `domain`, `service`, `status`, `owner`, `summary`, `target_repo`.
2. **List success — domains mode**: `mode`, `domains`, `total_domains`, `total_services`, `message`.
3. **Switch success**: all fields per tech-plan section 4.3 — `feature_id`, `domain`, `service`, `phase`, `track`, `priority`, `owner`, `stale`, `context_path`, `target_repo_state`, `context_paths`, `branch_switched`, `checked_out_branch`, `branch_error`.
4. **Switch failure**: `status: fail`, `error`, actionable `message`.

For each contract, document:
- Field names and types
- Nullability (which fields can be null and when)
- Stale warning behavior (threshold, flag, non-blocking)
- `exists` flag semantics in `context_paths`
- `branch_switched`/`branch_error` semantics

## Files

- `lens.core/_bmad/lens-work/skills/bmad-lens-switch/SKILL.md`

## Acceptance Criteria

- [ ] `list` features mode contract fully documented.
- [ ] `list` domains mode contract fully documented.
- [ ] `switch` success contract fully documented with all fields, types, nullability.
- [ ] `switch` failure contract documented with error codes and messages.
- [ ] Stale threshold (30 days), flag name, and non-blocking behavior documented.
- [ ] `target_repo_state` field documented: structure when populated, null when absent.
- [ ] `context_paths` field documented with `exists` flag semantics.
- [ ] Branch checkout result documented: `branch_switched`, `checked_out_branch`, `branch_error`.
