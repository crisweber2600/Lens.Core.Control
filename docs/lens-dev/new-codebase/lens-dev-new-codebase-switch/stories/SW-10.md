# SW-10 — Align Command Discovery Surfaces

## Context

Switch must appear consistently as `switch` across all four command discovery surfaces in the release module. Inconsistencies here make the command invisible or confusing to new users, and break agent-generated help menus.

**Carry-forward awareness:** M4 from finalizeplan-review — scope is bounded to switch-visible strings only; do not modify other command surfaces.

## Task

Audit and update the four discovery surfaces for consistency:

1. **Module help CSV** (`lens.core/_bmad/lens-work/bmadconfig.yaml` or `module-help.csv`): confirm `switch` is listed with description "Switch the active Lens feature context" (or aligned equivalent).
2. **Prompt manifest**: confirm `lens-switch.prompt.md` exists in the release module's prompts directory and is referenced in the manifest.
3. **Agent menu** (`AGENTS.md` or equivalent): confirm switch appears with correct invocation syntax.
4. **Published docs**: confirm the switch command entry in any public-facing command reference.

Run the module help discovery command to verify: the output must include `switch` as an available command.

## Files

- `lens.core/_bmad/lens-work/bmadconfig.yaml` (or module-help.csv)
- `lens.core/_bmad/lens-work/prompts/` (manifest or directory listing)
- `lens.core/AGENTS.md` (agent menu)
- `lens.core/_bmad/lens-work/README.md` or docs reference

## Acceptance Criteria

- [ ] Module help output includes `switch` with correct description.
- [ ] `lens-switch.prompt.md` exists in the prompts directory.
- [ ] Agent menu entry for switch present and syntactically correct.
- [ ] Published docs reference switch consistently.
- [ ] All four surfaces use the same command name (`switch`) and description.
- [ ] No non-switch command surfaces are modified.

## Dev Notes

- If the module help CSV uses a different format, follow the existing entry pattern for other retained commands.
- SW-12's string-scan regression should include a check that the module help CSV contains `switch`.
