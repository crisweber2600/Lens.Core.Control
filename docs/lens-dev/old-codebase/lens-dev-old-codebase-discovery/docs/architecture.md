## Architectural Overview

`lens.core.src` uses a **3-layer source-authoritative architecture**. It is the single place where all LENS Workbench module content is authored. When an IDE adapter is installed into a consumer control repo, it reads from this source; when the source is promoted, it becomes the distribution bundle (`lens.core.release`).


## Layer 2 Detail — `_bmad/lens-work/` (Source Module)

This is the only layer that lives in this repository. Everything authored here is the canonical upstream for both the release distribution and for IDE adapter generation.

### Module Identity Files

| File | Role |
| --- | --- |
| `module.yaml` | Canonical module identity: version `4.0.0`, skill list (41), prompt list (57), adapter config, dependency declarations |

| `module-help.csv` | Command discovery registry: maps commands to skills, descriptions, and quick-help entries |

### Dependencies (Declared in `module.yaml`)

| Dependency | Type | Notes |
| --- | --- | --- |
| `core` | Required | Core BMAD skills (not bundled — separate module) |
| `cis` | Optional | Creative Innovation Studio module |
| `tea` | Optional | Test Engineering Agent module |

> **Source ≠ Release:** The release distribution (`lens.core.release`) bundles all optional dependencies alongside `lens-work`. The source here declares them but does not bundle them. At install time, consumers choose which optional modules to include.

## Layer 3 Detail — Adapter Generation

The `_module-installer/installer.js` reads source files from `_bmad/lens-work/` and generates IDE-specific adapter layers into consumer control repos. It is **not** invoked inside this source repo directly — it is shipped as part of the install surface.

### Adapter Targets

| IDE | Output path (in consumer repo) | Entry point |
| --- | --- | --- |
| GitHub Copilot | `.github/` | `copilot-instructions.md`, `prompts/`, `instructions/`, `skills/` |
| Claude | `.claude/` | `CLAUDE.md`, `commands/` |
| Codex | `.codex/` | `AGENTS.md` |
| Cursor | `.cursor/` | `rules/` |

Adapters are **stubs** that reference the installed module. They are not the full SKILL.md content — they point the IDE to the correct module path.

## Authoring Scaffold Conventions

Because this is the source module, it enforces authoring conventions not present in the release:

### New Skill Convention

```
_bmad/lens-work/skills/bmad-lens-{name}/
├── SKILL.md                     ← required entry point
├── scripts/
│   ├── {name}-ops.py            ← operational script
│   └── tests/
│       └── test-{name}-ops.py   ← required test file
└── assets/
    └── {name}-template.md       ← template if needed
```

Plus a corresponding `_bmad/lens-work/prompts/lens-{name}.prompt.md`.

### Script Convention

All scripts use inline PEP 723 dependency declarations:

```python
# /// script
# dependencies = ["pyyaml>=6.0"]
# ///
```

Tests run via `uv run --with pytest pytest _bmad/lens-work/scripts/tests -q`.

## Promotion Pipeline (Source → Release)

```
lens.core.src (this repo)
       │
       │  1. Author + test locally
       │  2. Validate registry: validate-lens-bmad-registry.py
       │  3. PR to main
       │  4. CI: parity check (validate-lens-core-parity.sh)
       ▼
lens.core.release (separate repo)
       │
       │  beta branch = stable release
       │  Consumers: git pull origin beta
       ▼
Consumer control repos (installed via install.py)
```
