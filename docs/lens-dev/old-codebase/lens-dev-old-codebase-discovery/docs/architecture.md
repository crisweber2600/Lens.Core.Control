# lens.core.src — Architecture

## Executive Summary

`lens.core.src` is the source repository for the LENS Workbench payload. Its architecture is built around a strict separation between declarative contracts and imperative automation.

- `_bmad/lens-work/` contains the authoritative module implementation: skills, prompts, scripts, templates, contracts, and embedded documentation.
- The project exists to author, validate, and promote a toolkit. It does not run a service of its own.

## System Role

`lens.core.src` sits at the foundation of the control-repo lifecycle:

1. Contributors edit source in `lens.core.src`.
2. A promotion workflow builds a clean release payload and overlays the authored source assets.
3. A release repo receives the published payload.
4. Consuming control repos pull `lens.core/` and execute workflows from that downstream copy.

That makes `lens.core.src` the source of truth for both behavior and packaging.

## Architectural Layers

### 1. Module core — `_bmad/lens-work/`

The real product implementation. As of v4.0.0:

- `module.yaml` defines the module surface — version, published skills, prompts, scripts, dependencies.
- `lifecycle.yaml` defines the lifecycle contract — phases, tracks, and promotion rules.
- `skills/` (41 directories) hold the user-facing behavior model.
- `prompts/` (57 files) hold the authoritative prompt implementations.
- `scripts/` (19 files) hold operational automation — install, preflight, setup, PR, state derivation, registry validation.
- `assets/templates/` (11 files) hold planning and documentation templates.
- `docs/` (24 files) hold the embedded reference library.
- `_module-installer/installer.js` generates IDE adapter surfaces during release packaging.

### 2. Configuration and registry layer

The module relies on checked-in configuration instead of mutable application state.

- `_bmad/config.yaml` bridges Lens wrapper expectations to the module.
- `_bmad/lens-work/bmadconfig.yaml` is the authoritative config template.
- `_bmad/lens-work/module-help.csv` keeps the published command surface aligned.
- `_bmad/lens-work/assets/lens-bmad-skill-registry.json` maps published commands to skills.

### 3. Validation and contract layer

- `_bmad/lens-work/scripts/tests/` — 10 Python test modules for executable surface validation.
- `_bmad/lens-work/tests/contracts/` — 8 markdown contract references defining expected behavior for topology, provider adapters, governance, sensing, and planning handoffs.
- `scripts/validate-lens-bmad-registry.py` — cross-checks registry alignment.

## State Architecture

This project intentionally avoids service-backed runtime state.

- **Authoritative state** is held in git, YAML manifests, prompt files, skill files, registry files, and documentation.
- **Workflow state** for consuming control repos is reconstructed from lifecycle files and git primitives.
- **Release state** is produced by the promotion workflow and release branch history.

There is no database schema, no ORM layer, and no API server state to document.

## Execution Model

At runtime in a consuming control repo, the flow is:

1. A user invokes a published prompt or agent.
2. The adapter surface resolves to a module prompt, skill, or workflow.
3. Module manifests and lifecycle contracts determine routing.
4. Python scripts handle install, PR, validation, preflight, and other operational work.

`lens.core.src` authors and validates that flow before publication.

## Validation Strategy

Validation happens at four levels:

1. **Targeted pytest** files in `_bmad/lens-work/scripts/tests/`
2. **Contract references** in `_bmad/lens-work/tests/contracts/`
3. **Registry and surface consistency** via `validate-lens-bmad-registry.py`
4. **Control-repo preflight** after pulling the published downstream payload

## Design Constraints

- No executable product files outside approved script and installer locations.
- Packaging logic and module registry changes are coupled and should be changed together.
- Validation from the source root differs from validation from the outer control-repo root.
