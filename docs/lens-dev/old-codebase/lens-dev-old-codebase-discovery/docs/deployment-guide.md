# lens.core.src — Deployment Guide

**Date:** 2026-04-21
**Feature:** lens-dev-old-codebase-discovery
**Module Version:** 4.0.0

## What "Deployment" Means Here

`lens.core.src` does not deploy an application server. Its deployment path is source promotion into the release repository so control repos can pull the updated `lens.core/` payload.

## Promotion Trigger

The release path is defined in the source promotion workflow (in the full Lens.Core.Src workspace, this lives at `.github/workflows/promote-to-release.yml`). The workflow triggers on pushes that touch:

- `_bmad/lens-work/**`
- `.github/agents/**`
- `.github/prompts/**`
- `.github/skills/**`
- `.github/workflows/promote-to-release.yml`

## Promotion Flow

1. Checkout the source repository.
2. Read the module version from `_bmad/lens-work/module.yaml` (currently `4.0.0`).
3. Validate the source payload for required files and disallowed executable content.
4. Build a clean BMAD install into `build-output/`.
5. Overlay `_bmad/lens-work/` as custom module content.
6. Run `_bmad/lens-work/_module-installer/installer.js` to generate IDE adapters.
7. Overlay checked-in `.github/` source assets so authored stubs remain authoritative.
8. Validate the resulting payload and registry alignment.
9. Commit the payload to release `alpha`.
10. Open or reuse the `alpha` to `beta` pull request flow.

## Release Branch Model

- `alpha` — promotion target written by the workflow.
- `beta` — review target via pull request.

## Operational Checklist

### Before push

- Validate the specific scripts, prompts, or registries you changed.
- Confirm module manifests and published prompt/skill surfaces remain consistent.

### After push

- Wait for the promotion workflow to complete.
- Pull the downstream `lens.core/` repository copy in the control workspace.
- Run workspace preflight against the pulled release payload.

## Common Failure Modes

### Missing required module files

The workflow checks for core files: `lifecycle.yaml`, `module.yaml`, `module-help.csv`, `README.md`, and `_module-installer/installer.js`.

### Broken installer assumptions

Changes to `_module-installer/installer.js` or module registration can break the packaging step.

### Registry drift

The `lens-bmad-skill-registry.json` must stay aligned with `module.yaml` and the installer output.

## Verification After Promotion

From the outer control-repo root:

```bash
git -C lens.core pull
uv run ./lens.core/_bmad/lens-work/scripts/preflight.py
```
